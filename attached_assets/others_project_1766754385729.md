# OTA Upload API Server

A robust Node.js/Express server for managing Over-The-Air (OTA) firmware updates for IoT devices. This server handles firmware file uploads, version management, and provides intelligent update delivery to devices based on their current firmware version.

## 🎯 Overview

This OTA Upload API Server is designed to manage firmware updates for IoT devices (ESP32/ESP8266 or similar microcontrollers). It provides a centralized system for:
- Uploading firmware binary files
- Managing device-specific firmware versions
- Automatically serving firmware updates to devices
- Logging all OTA update activities
- Managing system error logs

## ✨ Features

### Core Features
- **Firmware Upload Management**: Upload `.bin` firmware files for devices
- **Version Control**: Track and manage firmware versions per device (by MAC address)
- **Smart Update Detection**: Automatically compare device firmware versions and provide updates only when needed
- **Device-Specific Firmware**: Support for device-specific firmware files or default fallback
- **Activity Logging**: Comprehensive logging of all OTA activities with timestamps
- **Error Log Management**: Centralized error logging with download/clear capabilities
- **File Management**: Delete firmware files when no longer needed
- **Authorization**: Simple password-based authorization for log access

### Security Features
- CORS enabled for cross-origin requests
- Helmet security headers (configurable)
- Authorization for sensitive log operations
- File upload validation

### Request Flow

#### OTA Update Request Flow
```
1. Device sends GET request → /otaFile/download?version=V1&deviceId=AA:BB:CC:DD:EE:FF
2. Controller extracts version & deviceId from query
3. Service queries database for device's target OTA version
4. Controller compares current version with target version
5. If versions match → Return 304 (No updates available)
6. If update available:
   a. Check for device-specific firmware: {deviceId}.ino.bin
   b. If not found, fallback to default.ino.bin
   c. Verify file accessibility
   d. Send firmware file for download
   e. Log the successful update
```

#### Firmware Upload Flow
```
1. Admin sends POST request → /otaFile/upload with file
2. Controller validates file presence
3. File saved to /uploads/{filename}.ino.bin
4. Return success response
```

## 🚀 Installation

### Prerequisites
- Node.js (v14 or higher)
- MySQL Database (v5.7 or higher)
- npm or yarn package manager

### Steps

1. **Clone the repository**
   ```bash
   git clone <repository-url>
   cd api-server-ota_upload
   ```

2. **Install dependencies**
   ```bash
   npm install
   ```

3. **Configure environment variables**
   Create a `.env` file in the root directory:
   ```env
   # Server Configuration
   HTTP_PORT=5001
   server_mode=development

   # Database Configuration (in config/config.js)
   DB_HOST=localhost
   DB_USER=root
   DB_PASSWORD=yourpassword
   DB_NAME=ota_database
   ```

4. **Setup Database**
   - Create MySQL database
   - Run the application once to auto-create tables (via Sequelize)
   - Or manually create the `ota_versions` table (see Database Schema)

5. **Create required directories**
   ```bash
   mkdir uploads log
   ```

6. **Start the server**
   ```bash
   npm start
   ```

   The server will run on `http://localhost:5001`

**Description:** Devices request firmware updates. Server checks version and sends update if available.

**Query Parameters:**
- `version` (required): Current firmware version of device
- `deviceId` (required): Device MAC address

**Example Request:**
```
GET /otaFile/download?version=1.0.0&deviceId=AA:BB:CC:DD:EE:FF
```

**Response (Update Available):**
- Binary file download (firmware)
- Logs: `[timestamp] Ota Update complete For :{comment} having [updated:{version}, mac:{deviceId},cat:{category}]`

**Response (No Update):**
```
304 No updates available
```

**Logic Flow:**
1. Query database for device's target `ota_version`
2. Compare with current `version`
3. If same → 304 response
4. If different:
   - Look for `{deviceId}.ino.bin` in /uploads/
   - If not found → use `default.ino.bin`
   - Send file for download
   - Log the update

**Use Case:** Device checks for updates on boot or periodically

## 🔄 How It Works

### OTA Update Workflow

#### Device Side (Arduino/ESP32 Code Example)
```cpp
void checkForUpdate() {
  String currentVersion = "1.0.0";
  String deviceId = WiFi.macAddress();
  
  HTTPClient http;
  String url = "http://yourserver:5001/otaFile/download?version=" + 
               currentVersion + "&deviceId=" + deviceId;
  
  http.begin(url);
  int httpCode = http.GET();
  
  if (httpCode == 200) {
    // Update available - download and flash
    int contentLength = http.getSize();
    bool canBegin = Update.begin(contentLength);
    
    if (canBegin) {
      WiFiClient * stream = http.getStreamPtr();
      size_t written = Update.writeStream(*stream);
      
      if (Update.end()) {
        Serial.println("OTA Success! Rebooting...");
        ESP.restart();
      }
    }
  } else if (httpCode == 304) {
    // No update available
    Serial.println("Already on latest version");
  }
  
  http.end();
}
```

#### Server Side Process

1. **Registration Phase:**
   - Admin registers device using `POST /otaFile/addOtaDetails`
   - Device MAC, comment, version, and category stored in database

2. **Firmware Upload Phase:**
   - Admin uploads firmware binary via `POST /otaFile/upload`
   - File stored as `{deviceId}.ino.bin` or `default.ino.bin`

3. **Version Management Phase:**
   - Admin updates target version using `PUT /otaFile/versionUpdate`
   - Database updated with new target version

4. **Device Update Phase:**
   - Device sends GET request to `/otaFile/download`
   - Server queries database for device's target version
   - Compares with device's current version
   - If different, serves firmware file
   - Logs update activity with timestamp

5. **Logging & Monitoring:**
   - All OTA activities logged to `ota.log`
   - Errors logged to `error.log`
   - Logs viewable/downloadable via API

### Logging System

**OTA Log Format:**
```
[2024-12-25 02:30:45 PM] Requested ota for :Living Room Sensor  [current:1.0.0, mac:AA:BB:CC:DD:EE:FF,cat:temperature]
[2024-12-25 02:31:12 PM] ### Ota Update complete For :Living Room Sensor having [updated:2.0.0, mac:AA:BB:CC:DD:EE:FF,cat:temperature] ###
```

## 🗄 Database Schema

### Table: `ota_versions`

```sql
CREATE TABLE `ota_versions` (
  `sl_no` INT AUTO_INCREMENT PRIMARY KEY,
  `mac` VARCHAR(255),
  `comment` VARCHAR(255),
  `ota_version` VARCHAR(255),
  `category` VARCHAR(255),
  `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  `updated_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

**Fields:**
- `sl_no`: Auto-increment primary key
- `mac`: Device MAC address (unique identifier)
- `comment`: Device description/location
- `ota_version`: Target firmware version
- `category`: Device category/type
- `created_at`: Record creation timestamp
- `updated_at`: Last update timestamp

**Example Data:**
```sql
INSERT INTO ota_versions (mac, comment, ota_version, category) VALUES
('AA:BB:CC:DD:EE:FF', 'Living Room Sensor', '2.0.0', 'temperature'),
('11:22:33:44:55:66', 'Bedroom Light', '1.5.0', 'actuator');
```
